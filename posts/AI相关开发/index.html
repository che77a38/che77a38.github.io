介绍几种联系上下文的方式

下面均以langchain为例

  1. Buffer Memory

    存储每段对话作为LLM的新输入

  2. Conversation Sumary Momory

    将对话信息传给LLM进行摘要,将此摘要作为开头部分传入LLM

  3. Conversation Buffer Window Memory

    将最近的k组对话存储起来,作为下一轮的LLM的输入

  4. Conversation Summary Buffer Memory

    第2种和第3种方式的结合,将最近的k组对话存储起来以及[上一轮的摘要+老k组对话]的摘要+新k组对话,作为下一轮的LLM的输入(使用token长度来决定什么时候刷新输入)

    优点,不仅有短期记忆,也有长期记忆

  5. entity memory

    通过LLM提取对话中有关实体的知识并积累

  6. vector store back memory或思维导图记忆方式

    向量数据库匹配或思维导图记忆方式

对长文本做总结的方式

下面均以langchain为例

  1. stuff模式

    直接将document传给LLM做总结

  2. map_reduce模式

    分割多个片段,通过LLM对每个片段进行总结,再将总结拼接再交给LLM总结

  3. refine模式

    分割多个片段,先对第一个片段进行总结,再将前一个片段的总结加上下一个片段进行总结,以此总结递推直到只有一个总结

    优点:增加了总结内容的连贯性

  4. map_rerank模式

    一种搜索答案的匹配方式

    根据一个问题,给每个document片段计算一个能回答这个问题的概率分数,然后找到分数最高的document片段,将其转换为问题的一部分,加问题发送给LLM

  • 想获得更综合性的知识:使用refine或map_reduce模式
  • 获取更精确的细节知识:使用map_rerank模式

litellm

自动负载均衡：当请求发送给 minimax-m2.1 时，LiteLLM 的 Router 会发现数据库里有两个 ID 为 minimax-m2.1 的后端配置，它会自动在两者之间进行负载均衡

容器化部署

负载均衡配置多模型优先级

  1. 在 Models + Endpoints 列表页。
  2. 找到你的 minimax-m2.1 模型（也就是你刚才改名的那两个），点击右侧的 编辑图标（✏️ 铅笔形状）
  3. 在弹出的编辑窗口中，寻找名为 “Model JSON”、“Advanced” 或者 “Configuration” 的区域
    • LiteLLM UI 的不同版本界面略有不同，但通常都会提供一个可以直接修改 JSON 的输入框
  4. 在 JSON 配置中，找到 litellm_params 字段（如果没有，就加在最外层，或者确保它在参数对象里）
  5. 加入 "priority": 0

当一个请求进来时，LiteLLM 的 Router 会按照以下步骤执行：

  1. 第一层：优先级筛选 (Priority Check) 系统首先检查所有叫 minimax-m2.1 的模型。它会锁定在当前可用的最高优先级（比如 priority: 0）的那一组模型上。
    • 此时，你的 OpenRouter 模型（priority: 1）直接被排除在外，根本不参与接下来的负载均衡计算。
  2. 第二层：策略选择 (Routing Strategy) 在锁定的这组 priority: 0 的模型（即你的两个内网模型）中，系统会根据你配置的 least-busy 策略进行选择。
    • 系统会比较这两个内网模型当前谁正在处理的请求更少，然后把新请求发给比较“闲”的那个。

故障切换 (Failover)

  • 如果内网模型 A 和 B 都挂了（比如超时、返回 500 错误），LiteLLM 将它们标记为 Unhealthy。
  • 此时 Priority 0 组全军覆没。
  • Router 降级 到 Priority 1 组。
  • 结果：请求发给 OpenRouter。

上下文工程

AI本身是没有记忆的,但他每次收到的信息都是完整的对话,所以看上去就像是有记忆一样

上下文Context: 这个被一次性发给AI的完整的历史记录

上下文工程Context Engineering: 如何管理和修改这段历史记录的技巧

普通的网页聊天只需要简答的上下文工程

但当面对agent这种可以加载工具的ai对话的情况下,大量的ai工具调用会很容易使得大语言模型跑偏,忘记自己到底要干什么

所以如何通过一套程序化的规则,来自动管理和修改这个上下文,确保AI在漫长的自主行动中,始终符合用户的最初要求,这就是上下文工程要解决的核心问题

目前没有一套公认的完整方案,但是下面会罗列当前常用的方法

  • 让AI学会记笔记

    本质:人类用户通过控制最初的提示词来指导AI如何写笔记,再通过把笔记放到上下文中最显眼的位置(transform架构的AI大模型的上下文中首尾位置就是最显眼的位置),来间接引导整个AI的处理流程

    此时的提示词长这样👇🏻

  • 直接丢掉太老的消息/压缩太老的信息,只保留新消息,当然最开始用户和系统提示词是必须要保留的

    • 把太老的信息,让AI模型去总结其中的关键信息,再用这个精炼的总结去替换掉原来的上下文从而达到压缩的目的
    • 除了👆🏻,还有更高级的压缩方法,ToolResponse可能会非常长,比如包含一篇非常长的文章,这时候agent就会通过RAG技术把这个文章处理后存到一个临时的向量数据库中,然后修改ToolResponse不再包含原文,而是用一句话来代替,如文章已经存入知识库,数据用query_doc工具查询
    • 还有的方法是直接优化工具的返回值,对于网页浏览工具,可以先去掉网页中不必要的html标签,只返回最核心的内容给AI模型,从源头减少信息冗余

    • 作用：为AI助手添加类似MemGPT的长期记忆能力。让AI能记住之前的对话历史、用户偏好和上下文，实现更连贯、个性化的交互，而不仅是单次对话。
  2. Replicate
    • 作用：让AI助手能够直接调用云端数千个开源AI模型（如图像生成、语音识别、大语言模型等），无需本地部署，扩展了AI本身的多元处理能力。
  3. AI Research Assistant / AI搜索
    • 作用：专门增强AI的搜索和研究能力，可能整合了学术数据库、网络搜索或特定知识库，帮助开发者或研究人员快速获取和总结技术信息。

这类MCP让AI能直接操作开发和云原生工具链。

  1. GitHub / GitLab MCP
    • 作用：核心的代码仓库操作。允许AI助手直接读取仓库文件、代码，甚至执行创建分支、提交代码、管理Pull Request、查看提交历史等Git操作。是将AI深度集成到开发流程的关键。
  2. Vercel / Cloudflare Pages MCP
    • 作用：前端部署自动化。AI可以触发构建、部署网站或应用，管理生产/预览环境，实现从代码到上线的自动化。
  3. Databutton
    • 作用：可能让AI能够构建和部署数据驱动的Web应用（如内部工具、仪表盘），将想法快速转化为可用的应用。
  4. Neon / Supabase MCP
    • 作用：数据库操作。允许AI安全地连接并操作数据库（PostgreSQL），执行查询、管理数据，将AI与动态数据源连通。

claude.md

它放在项目根目录，Claude Code 每次启动都会先读它，相当于你给新来的同事写的项目交接文档，区别是它每次都会从头认真读一遍而且严格执行

写得好不好，三件事最关键。写得短一点，150 行以内为佳，写太长会挤压后续对话的空间。语气直接，用命令式，别写”我们团队比较喜欢”这种软话，”所有注释用中文”比”团队偏好中文注释”有效太多。每条都能判断，”代码质量要高”没用，”函数超过 50 行必须拆分”才能落地。

四条最值钱的规则，直接拿去用：先问清楚再动手、简单优先、只动该动的、做完要验证。展开就是：别让它猜你的意图，目标说清再写；能两行解决的不写两百行，拒绝过度设计；不要顺手重构没让它改的代码；跑通构建和测试才算完，没通过别说完成。

业务场景的需求，光描述功能还不够。开头先把问题写清楚，要解决什么、给谁用、怎么算做对了，别一上来就列功能清单。比如说我们要写一个国际门票频道页时，第一句话就是”国际门票目前没有独立入口，用户只能搜索找到，非热门城市曝光极低”，这两句话决定了它后面碰到”热门城市展示几个”“筛选要不要做’最近浏览’“这类问题时的判断方向。

接下来要给它划范围。Claude Code 很积极，你说做一个列表页它顺手就给你加上收藏、分享、埋点。明确写出”不做登录态、不做分享、不做 SEO，下一期再说”，它就不会越界。异常情况要单独列出来，接口超时怎么办、数据为空展示什么、图片挂了用什么兜底，这些不写它要么不处理，要么猜个你不满意的方案。

验收标准必须给数字，”页面要快”没用，”首屏 1.5 秒内”才能判断；”布局正常”没用，”在 375 和 1440 两个宽度下不错位”才能验收。

写需求的时候，别用”待定”“后续再看”“TBD”。Claude Code 碰到这些会自己猜着填，猜的往往不是你要的。哪怕写”这一版先硬编码，下版再做配置化”，也比空着强。

怎么确认它真的做对了

它跟你说”搞定了”其实没用，关键是你怎么验收，因为它也会用最省事的方式交差

我自己就看三件事。命令过没过，构建和测试跑完绿灯就行，CLAUDE.md 里写好”完成后跑 make build && make lint“它会自己做。眼见为实，页面打开看一眼、数字对一下、关键流程试一下，改完文件不代表页面就是你要的样子。对照清单，需求里写好的验收标准一条一条过，没对完不算完成让它接着改。

改坏了怎么救回来

不会写代码的人最怕代码被改乱了找不回来，常用的就两条。

Git 快照，每次大改前让它先跑一遍 git status 看清楚都有什么，确认没问题再让它 commit 一个检查点。改坏了直接说”按刚才的检查点回退”，比自己手动 checkout 安全得多。

撤销上一步，直接对它说”撤销刚才所有改动”，或者按 /rewind 回到上一个状态。

别让它陷入改了试试的死循环

有个坑很容易踩：陷入改了试试的循环，4-5 轮下来本来不大的问题变成一团乱麻。原因就一个，没诊断清楚就开始打补丁。

避免方法也一句话：根因没说清楚之前先别动代码。让它先答”问题出在哪个文件的哪一行，为什么会这样”，答含糊继续查，答清楚再改。一上来说”我试试改 X 看行不行”的，直接喊停让它先答根因。

长会话怎么办

Claude Code 的上下文容量是固定的，跑久了早期内容会被挤出去

任务做完就 /clear，一个会话只做一件事，做完清掉再开下一件，两件不相干的事在同一个上下文里来回切，它会越做越乱。

长任务结束前让它写交接笔记，直接对它说：”把当前进度写成一份 HANDOFF.md，包括做了什么、试过什么没成功、下一步该做什么。” 第二天打开新会话，把这个文件给它，就能接着干，不依赖任何压缩算法。

其他细节

截图比文字快，要描述一个界面问题或者想参考某个设计风格，直接丢图比写一段话准多了，布局、颜色、层级都带进来了，让它少猜

任务拆小一件件来，一句话能讲清楚的任务它几乎不会出错。一上来给一大坨需求，它中间任意一步走偏后面就全偏了，一件做完验收一件再开下一件。

对话跑偏了就重启，在已经跑偏的对话里来回纠正它越纠越乱，清掉上下文重说一遍需求往往更快。第二天接着干，先翻一眼上次的 Recap（/clear 后会自动生成的会话摘要）想起来干到哪了。

Memory 跨项目记住你的偏好，CLAUDE.md 是项目级的每个项目都得单独写一份，Memory 是用户级的跨所有项目和会话都生效。直接对它说”记住我喜欢先看方案再执行”、”记住回我中文”，它会写进 ~/.claude/memory/，以后任何项目打开都记得。常交代的背景信息都可以沉淀进去，省得每次重复说。

双击 ESC 改上一条，说错了或者它跑偏了，按两下 ESC 就能回到上一条消息修改，不用重开会话。

自建skill

Skill 本质就是一个文件夹，放在 .claude/skills/ 目录下，里面有个 SKILL.md 写清楚什么时候用、要做什么。Claude Code 启动时只读 frontmatter，也就是描述触发条件的约 100 个字，真正调用时才加载完整内容，所以装几十个 Skill 启动也不会变慢

用得最多的有三种类型Skill

工作流型

把每次都要做的固定步骤打包。比如整理周会纪要

检查清单型

上线前、发版前、提交前过一遍，避免漏项。比如需求上线检查：

领域专家型

把判断框架沉淀进去，碰到这类问题按固定路径走，不让它每次自由发挥。比如线上问题排查，以及你们平时业务最佳实践的 SOP

写好之后，把它放到 .claude/skills/ 文件夹下，碰到对应场景说一句”用整理周会”或”用线上排查”就行，这里你也可以让 Claude 帮你写。此外两个写 Skill 的小坑要避一下。

description 写触发条件，不写功能介绍，”开完会有原始记录需要整理时调用”比”把会议录音整理成结构化周报”准确率高得多。

一个 Skill 只做一件事，别把审查、发布、调试塞在一起，拆开用起来才更准。

Engineering工程类

User-invoked

  • ask-matt — 路由器，问”我该用哪个 skill 或流程”。对 repo 里的 user-invoked skill 进行路由。
  • grill-with-docs — 严格质询 + 同步建领域模型，更新 CONTEXT.md 和 ADR。
  • triage — 把 issue 沿着状态机推进。
  • improve-codebase-architecture — 扫代码库找 deep module 机会，生成可视化 HTML 报告，然后质询选中的机会。
  • setup-matt-pocock-skills — 配置 repo 使用 engineering skill（issue tracker、triage labels、domain doc 布局）。每个 repo 用其他 engineering skill 前跑一次。
  • to-spec/to-prd — 把当前对话提炼成 spec，发布到 issue tracker。无需访谈，只是综合已讨论的内容。
  • to-tickets/to-issues — 把任何计划、spec 或对话拆成 tracer-bullet tickets，每个声明 blocking 关系——写成本地文件或真实 tracker 的原生 blocking 链接。
  • implement — 按 spec 或 tickets 实现，在约定的 seams 预埋 /tdd，提交前调 /code-review 收尾。
  • wayfinder — 规划跨多次会话的大块工作，做成 issue tracker 上的 investigation tickets 共享地图——一个个解决直到路清晰。

Model-invoked

  • prototype — 一次性原型验证设计问题——CLI 验状态/逻辑问题，或多个可切换的 UI 变体。
  • diagnosing-bugs — 难 bug 和性能回归的调试循环：复现→最小化→假设→埋点→修→回归测试。
  • research — 查一手资料，输出带引用的 Markdown，放后台跑。
  • tdd — 红绿重构循环，垂直切片式实现功能或修 bug。
  • domain-modeling — 主动构建和打磨项目领域模型——用 glossary 挑战术语，用边缘场景压力测试，更新 CONTEXT.md 和 ADR。
  • codebase-design — 设计 deep module 的共享规范和词汇：行为 behind 小接口，放在干净的 seam，通过接口测试。
工作流类工作流类

User-invoked

  • grill-me — 严格质询计划或设计，直到决策树每个分支都覆盖。
  • handoff — 把当前对话压缩成 handoff doc，让其他 agent 接着干。
  • teach — 多会话教学，用当前目录作为有状态工作区。
  • writing-great-skills — 写/编辑 skill 的参考：让 skill 可预测的词汇和原则。

Model-invoked

  • grilling — 严格质询计划或设计，直到决策树每个分支都覆盖。/grill-me 和 /grill-with-docs 背后的复用循环。
正式工作流

  1. 仓库初始化（仅新项目 / 新接入仓库执行 1 次）

    /setup-matt-pocock-skills完成仓库标准化配置

  2. 需求澄清与标准化文档产出

  • 已有业务文档 / 历史讨论

    /grill-with-docs 具体需求xxxxx描述清楚需求

  • 仅有口头 / 对话需求，无书面文档

    1. /grill-me（人工质询全分支，覆盖所有异常、约束、性能、兼容需求）
    2. /to-spec 把对齐后的完整对话提炼为正式需求规范，自动同步至工单系统存档。

  1. 代码库架构分析与长期工作规划

    /improve-codebase-architecture 扫描存量代码库，生成可视化 HTML 架构报告，识别深模块拆分、解耦、分层优化机会

    /wayfinder 生成全局调研工单地图，按依赖拆分多轮会话任务，逐个扫清未知技术卡点，直至整体路径清晰

  2. 需求拆解为可执行工单

    /to-tickets 基于 Spec + 架构报告，拆分可执行最小工单，自动标注上下游阻塞依赖，支持输出本地文件或工单系统原生关联链接。

  3. 功能开发落地

    /implement 按工单 / Spec 启动实现流程，预埋 /tdd 钩子；

  4. 工单流转、交接、沉淀规范

    /triage 驱动所有工单沿状态机流转：待调研→待开发→编码中→待评审→测试→上线→归档；

  5. 跨会话 / 跨 Agent 工作移交

    /handoff 压缩当前全部上下文、Spec、工单、架构文档生成标准化交接文档，供其他 Agent 承接

  6. 沉淀技能编写规范（流程复盘后按需执行）

    writing-great-skills 梳理本次流程中好用的 Skill 使用范式，沉淀可复用编写 / 调用原则

技能路由查询（任意阶段遇工具选择困惑）ask-matt 输入当前工作目标，自动推荐匹配 Skill 与执行顺序

线上 Bug / 性能问题修复闭环

  1. ask-matt（确认执行路径）
  2. triage（更新工单状态）
  3. grill-with-docs（核对相关业务领域边界）
  4. to-tickets（拆分缺陷修复子工单）
  5. implement
  6. triage（推进工单至完成状态）

Agent Skills技术详解

一种提示词工程技术，通过规范化的文档（Skill）让AI（如Claude Code）掌握特定领域的技能. 免用户每次重复输入冗长要求，提升AI执行任务的准确性和效率

目标：避免用户每次重复输入冗长要求，提升AI执行任务的准确性和效率。

技术核心机制

渐进式加载（节省Token）：

  • 元数据固定加载：仅向AI传递Skill的名称和描述。
  • 指令触发时加载：当AI识别任务需要某Skill时，才请求其具体内容。
  • 资源按需加载：关联文件（如CSS配置、Python脚本）在需要时动态调用。

Skill的层级结构：

  • 主文件（如Skill.md）可引用子文件（如“科技风.md”），避免单文件过长。
  • 支持复杂工作流（如通过CSV表格管理UI组件，配合脚本动态检索）。

与MCP技术的关系

  • Agent Skills 和 MCP（Model Context Protocol） 均属提示词工程的规范化实践，前者侧重技能封装，后者侧重工具调用。
  • 两者可结合使用，实现更复杂的AI智能体行为（如动态调用外部API）。

RAG

Retrieval Augmented Generation

给予大模型参考文档,让大模型只发相关回答

传统RAG

需要一种新的模型,Embedding模型,和大语言模型不同,它的作用是将一段文字输出成一个固定长度的数组

可以理解成对原始内容的有损压缩

embedding模型会让意思越相近的文字,位置越相近

在发给AI大模型问答的时候,首先会将问题转换为数组,找临近的文字片段作为语义片段,一并发送给大模型来让他回答,这样就会回答更相关的内容了,产生幻觉的风险也就小了很多

如果文档太长了,还有一道工序叫Chunking,切块,切块后全部经过embedding转换为向量,然后还要把每个向量和原始文本片段的对应关系保存起来,这就是向量数据库

RAG的缺陷

  • 分块的过程中很容易丢失指代语义等块与块之间的关系,导致回答牛头不对马嘴

    临时解决方案就是在预处理的时候把指代语义全部换成主体名,或者让大模型参与分块的过程,根据语义自动判断应该在哪里断

  • 全局问题和局部片段的匹配度太低: 缺乏全局视角,比如说文章里一共出现了多少个’我’字,RAG对于这种问题,无能为力

目前来说,还没有哪个方案是真正十全十美的

GraphRAG

用知识图谱来解决传统RAG的问题的

GraphRAG的每一步都离不开大语言模型的参与

这种既有实体,又有关系,还可以加属性的图结构叫做LPG(Labeled Property Graph)

如何建立知识图谱

大语言模型出现之前,构建知识图谱的算法,晦涩难懂又效果很差

步骤

  1. 命名实体识别(Named Entity Recognition)

    识别”实体”,类型包括”人”,”物体”,”味道”

  2. 关系抽取(Relation Extraction)

    识别”实体”间的”关系”

这两任务,现在仅仅需要一句prompt就搞定了

再定义返回格式

每次生成完图谱后,还要将图谱连同原文再发给AI一次,问他是否还有遗漏的,如果说有,就再补一轮,这个过程循环直到AI说没有

这个反复盘问榨干信息的过程名叫Data Gleaning

于是GraphRAG会对文章中的每段文字都单独生成一个知识图谱

等所有的片段都处理完之后,GraphRAG会把名字相同的实体自动进行合并,最后会拼成一个完整的文章级的知识图谱,这个过程不需要AI参与

最后GraphRAG会将每个被合并的描述信息交给大语言模型,请他根据这些信息生成更通顺的总结性描述,并用它来当做合并后的最终描述

p.s.一个细节是GraphRAG一直在维护着知识图谱和原文之间的对应关系

GraphRAG知道图谱中的信息是由哪几段原文生成出来的,反过来,GraphRAG知道任一段原文在知识图谱里面对应了哪些实体哪些些关系

到这查询起来还是不够方便,因此还有一步操作: GraphRAG会让AI使用每个子图中的点和边的描述信息,生成一个更高级的总结性描述

如果图还不够简单,就再合并,再总结,一层层向上,最终形成一个知识图谱的层级结构

最后这个知识图谱的层级结构变成一个能被快速检索的向量数据库

把知识图谱中的每个节点,每条边的实体信息和总结性描述都当做一个文字片段进行embedding,然后存入向量数据库,再加上原文中的每个切片也都存入向量数据库,这就是知识图谱的最终形态了

如何查询知识图谱

用户问的问题转换为向量,在向量数据库进行匹配

然后GraphRAG会将所有找到的这些点和边的总结性描述以及他们关联的原文片段再加上用户提的问题,一起打包发给AI大模型

这样无论问的是一个高层次的抽象问题,还是某段原文的细节问题,相关信息都会被一并带上
