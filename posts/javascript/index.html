javascript及typescript语法盘点,由C#开始理解

javascript

以 C#/C++ 知识做对照，规避踩坑，快速掌握 ES6 + 现代 JS

  1. 浏览器：网页交互，提供 DOM/BOM/WebAPI

隐式类型转换

JS的大坑,C#/C++会编译报错,JS静默转换

  • 1 + "2" → "12"：数字 + 字符串 = 字符串拼接，C# 编译报错，JS 静默转换
  • 1 == true → true：== 自动类型转换，=== 严格相等（不转换）
  • 0 == false → true：假值宽松相等，=== 结果为 false
  • "" == false → true：空字符串宽松等于 false，=== 结果为 false
  • "0" == false → true：字符串 "0" 宽松等于 false，=== 结果为 false
  • null == undefined → true：两者仅宽松相等，=== 结果为 false
  • null == 0 → false：null 不与数字转换相等，仅等于 undefined
  • undefined == 0 → false：undefined 不与数字转换相等
  • [] == false → true：空数组隐式转换为假值
  • [] == ![] → true：先取反再比较，双重隐式转换经典坑
  • {} == {} → false：引用类型比较内存地址，两个对象永不相等
  • [] + [] → ""：数组相加隐式转换为空字符串
  • [] + {} → "[object Object]"：数组与对象拼接为字符串
  • {} + [] → 0：{} 被当作空代码块，+[] 转换为数字 0
  • "5" - 3 → 2：减号触发字符串强制转为数字运算
  • "5" + 3 → "53"：加号触发数字强制转为字符串拼接
  • null + 1 → 1：null 隐式转换为数字 0
  • undefined + 1 → NaN：undefined 隐式转换为 NaN

C# 只有 bool 做条件，JS 任意值都能布尔化

原型

JS 是基于原型的语言，所有对象都有一个隐藏的原型对象，方法、属性共享在原型上，节省内存

关键连接

  • 实例对象：obj.__proto__ → 指向原型对象
  • 构造函数：Fn.prototype → 指向原型对象
  • 原型对象：.constructor → 指回构造函数

三条铁律

  1. 所有函数都有 prototype（原型对象）
  2. 所有对象都有 __proto__，指向构造函数的 prototype
  3. 实例共享原型上的方法，不用每个实例创建一次，极大节省内存

原型利用方式

class只是语法糖

底层依然是原型，不是真正的类继承

总结

  • 函数.prototype = 原型对象（存放共享方法）
  • 对象.__proto__ = 指向原型
  • 实例→原型→原型的原型→… = 原型链
  • 优点：共享方法、节省内存、实现继承
  • 特点：JS 独有，C#/Java 没有原型概念
原理

V8底层可运行模拟代码

底层就是 C++ 指针！实例通过指针指向公共原型对象，实现方法共享、节省内存

原型到底是什么？

原型 = 被指针指向的「普通对象」

没有任何魔法，就是一个存共享方法 / 属性的公共对象。

  • 同一个构造函数 / 同一个 class 创建的所有对象，它们的隐藏指针 __proto__ 指向 完全同一个 原型对象！
  • 不同构造函数创建的对象，指向不同的原型对象。

JS 世界的本质：万物皆对象，靠原型指针关联共享属性和方法

闭包Closure

内层函数记住并访问外层函数作用域，即使外层函数已执行完毕，变量依然保留。

形成条件

  1. 函数嵌套
  2. 内层函数使用外层函数变量
  3. 内层函数被外部引用

执行过程：

  1. createCount() 执行，创建 count=0
  2. 返回内层函数，带走作用域
  3. 外层函数执行完毕，但 count 不销毁
  4. 每次调用 cnt() 都操作同一个 count
  5. count 成为私有变量，外部无法直接修改

闭包优点

  • 私有化变量，避免全局污染
  • 持久化保存状态（计数器、缓存）
  • 函数式编程核心

闭包缺点

  • 变量不销毁，过多会造成内存占用
  • 不需要时及时释放引用
原理

JS 闭包在 V8 引擎底层 = 函数对象 + 堆上 Context 对象 + outer 指针链

V8 用以下核心结构实现闭包

闭包创建全流程（C++ 视角）

当 JS 执行function outer() { let a=1; return function inner() { return a; } }时，V8 做了这些事

步骤 C++ 引擎动作 内存变化
1 预解析 inner，识别引用的自由变量a 标记a需被捕获
2 outer 执行时，在堆上创建 Context 对象 堆内存分配 Context，存入a=1
3 inner 函数对象的context_指针指向这个 Context JSFunction→Context（堆）
4 Context 的outer_指针指向 outer 的外层 Context 形成作用域链
5 outer 执行完毕，栈帧销毁 但 Context 因被 inner 引用而保留在堆
6 inner 被调用时，顺着context_→outer_链查找变量 实现跨作用域访问

为什么栈变量能 “活下来”？

  • 栈帧销毁不影响堆：outer 的栈帧（局部变量区）被销毁，但捕获的变量已被复制到堆上的 Context 对象

    闭包捕获 = 直接持有变量本身的引用 / 绑定（这才是本质）

    情况 1：变量没有被闭包捕获

    • 引擎直接把变量存在 栈内存
    • 函数执行完，栈销毁，变量一起销毁

    情况 2：变量被闭包捕获

    1. 引擎在编译阶段就扫描到：这个变量被内层函数引用了
    2. 引擎直接跳过栈，把这个变量创建在堆的 Context 对象中
    3. 栈上根本就没有这个变量！
    4. 函数内外访问的，都是堆里的同一个变量

    变量被回收取决于V8 GC,它的核心判断规则：

    • 可达性：从「根对象」(全局对象、调用栈) 能通过引用链找到的对象，视为「活跃」，不回收
    • 不可达性：当没有任何引用指向闭包函数时，整个引用链断裂，Context 及内部变量被标记为「垃圾」，最终被回收

  • GC 判断可达性：只要 inner 函数存在，Context 就被引用，不会被垃圾回收

使用闭包的时候要格外注意: 避免泄漏：及时清理闭包引用，警惕全局 / 事件 / 定时器持有闭包

异步编程

  • C#：async/await + Task；
  • JS：回调→Promise→async/await 三代演进

回调

异步操作以来上一个异步的结果,层层嵌套回调函数,代码横向扩张

核心缺点: 可读性极差,难以维护,错误捕获混乱,无法统一处理异常

本质: 无规范的异步嵌套,对标C#淘汰的嵌套回调写法

规范：禁止使用，全部替换为 Promise /async/await

Promise

ES6 标准，封装异步，对标 C# Task

Promise 是一个异步操作容器，管理异步的成功 / 失败状态，解决回调地狱

三种状态（状态一旦改变，永久凝固，不可修改）：

  1. pending：进行中（初始状态）
  2. fulfilled：成功（调用 resolve）
  3. rejected：失败（调用 reject）

核心方法

  • .then()：成功回调，可链式调用（返回新 Promise，彻底解决回调地狱）
  • .catch()：统一捕获所有异步错误（推荐写在最后）
  • .finally()：无论成功 / 失败，最终必执行（用于清理操作）

链式调用（替代回调地狱）

async/await

ES2017，现代标准，和 C# 语法几乎一致

  • Promise 的语法糖，让异步代码写起来像同步代码，前端最推荐写法
  • 对标 C#：async/await + Task，语法、逻辑、用法完全对齐

语法规则

  • async：修饰函数，标记该函数为异步函数，返回值自动包装为 `Promise
  • await：只能写在 async 函数内部，暂停执行，等待 Promise 执行完成
  • 异常处理：必须用 try/catch 捕获（对标 C# 异常捕获）

串行异步

事件循环（EventLoop）

JS 独有

JS 是单线程语言，事件循环是实现「异步非阻塞」的底层核心机制，完全区别于 C# 的多线程 / 线程池模型

  • JS 只有一个主线程

    所有代码（同步、DOM 渲染、异步回调）都在唯一主线程执行，绝对没有并行执行的 JS 代码。

  • 主线程不能阻塞

    一旦主线程卡住（死循环 / 长时间计算），页面会卡死、无法点击 / 渲染。

  • 异步任务绝不阻塞主线程

    定时器、网络请求、文件操作等耗时任务，不占用主线程，由浏览器 / Node 底层的 C++ 线程池执行，完成后通知 JS

两大任务队列

异步任务分为宏任务和微任务，有严格的执行优先级：

  1. 宏任务（Macrotask）
  • 代表：setTimeout/setInterval、AJAX / 网络请求、script 标签代码、IO 操作
  • 特点：每次只执行一个，执行完就去清空微任务
  1. 微任务（Microtask）
  • 代表：Promise.then/catch/finally、async/await、queueMicrotask
  • 特点：一次性清空所有，优先级远高于宏任务
事件循环执行铁律（固定顺序）
  1. 执行主线程所有同步代码（立即执行，优先级最高）
  2. 同步代码执行完毕 → 清空当前所有微任务（全部执行完）
  3. 微任务清空 → 执行一个宏任务
  4. 宏任务执行完 → 再次清空微任务 → 执行下一个宏任务
  5. 循环往复 → 这就是事件循环

开发踩坑盘点

  • ❌ 变量无固定类型，不要用 C# 静态思维定义变量
